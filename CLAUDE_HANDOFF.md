# JLOA 프로젝트 진행상황 및 AI 인수인계

> 기준일: 2026-07-17
>
> Claude와 Codex는 작업을 시작하기 전에 이 문서와 실제 코드를 함께 확인한다.
>

## 1. 저장소와 기준 코드

| 영역 | GitHub | 현재 작업공간 | 기준 브랜치 |
|---|---|---|---|
| 웹 | `https://github.com/ION127/jloa-web.git` | `.jloa-web-push` | `main` |
| 백엔드 | `https://github.com/ION127/jloa-backend.git` | `lostark_backend` | `main` |

- 웹의 배포 기준은 `jloa-web` 저장소다.
- `lostark_app/website`는 현재 IDE에서 확인하기 위한 로컬 사본이다. 웹 변경 후 필요한 파일만 동기화한다.
- `lostark_app` 상위 작업공간에는 다른 기능의 미커밋·미추적 파일이 많다. 웹 작업을 이유로 그 파일을 정리하거나 함께 커밋하지 않는다.
- 백엔드 `main`에는 같은 서버 원정대의 이름·직업·전투력을 제공하는 요약 API가 포함되어 있다.

## 2. 현재 사용자 흐름

1. 사용자는 `/index.html`에서 JLOA의 주요 기능을 한눈에 본다.
2. 메인 상단의 캐릭터 검색창에 닉네임을 입력한다.
3. 프론트는 `/character.html?name={닉네임}`으로 이동한다.
4. 캐릭터 페이지가 `https://api.jloa.cloud/api/character/{닉네임}`을 호출한다.
5. 사용자가 `최신 정보 갱신`을 누르면 같은 API에 `?fresh=true`를 붙인다.
6. 백엔드는 로스트아크 Open API를 조회하거나 SQLite 캐시를 반환한다.

메인 페이지는 검색창만 있는 페이지가 아니다. 기능 소개를 유지하면서 가장 자주 쓰는 캐릭터 검색을 눈에 띄게 제공하는 것이 확정된 방향이다.

## 3. 웹 구현 현황

### 메인 페이지

- `index.html`: 검색 중심 메인과 기능 카드
- `web-app.js`: 닉네임 검증, 최근 검색 5개 저장, 캐릭터 페이지 이동
- `web-app.css`: 메인 전용 레이아웃
- `desktop-app.html`: 데스크톱 앱 소개와 다운로드
- 개인정보 처리방침, 광고 정책, 릴리스 노트 페이지가 분리되어 있다.

### 캐릭터 페이지

`character.html`은 메인과 분리된 전용 페이지다. 현재 세 탭으로 구성한다.

- `캐릭터 정보`
  - 전투 특성
  - 카드
  - 아크 그리드
  - 각인
  - 보석
  - 장비·액세서리·특수 장비
  - 아크 패시브
  - 전투 스킬과 트라이포드
- `원정대`
  - 검색한 캐릭터와 같은 서버에 속한 원정대 캐릭터만 표시
  - 캐릭터 이름, 직업, 전투력 표시
  - 현재 검색한 캐릭터 강조
  - 캐릭터 행 클릭 시 해당 캐릭터의 기본 `캐릭터 정보` 화면으로 이동
  - 탭을 처음 열 때만 백엔드 요약 API를 지연 호출
- `기타 정보`
  - 캐릭터 프로필
  - 성향
  - 장착 아바타
  - 수집품

상세 정보가 있는 장비, 보석, 각인, 코어, 카드, 스킬, 아바타는 클릭 시 공통 `dialog`에서 세부 내용을 보여준다.

### 현재 디자인 원칙

- Loawa와 Lopec의 정보 구조는 참고하되 코드, 이미지, 문구, 독점 계산식을 복제하지 않는다.
- 한 화면에서 비교하기 쉬운 조밀한 대시보드를 유지한다.
- 정보 유형을 탭으로 분리하고 탭 클릭 시 페이지 내부 위치로 스크롤시키지 않고 해당 패널만 표시한다.
- 960px 이상에서는 좌측 프로필과 우측 요약표를 나란히 표시한다. 960~1179px은 3개 요약열, 1180px 이상은 4개 요약열을 사용한다.
- 모바일 620px 이하는 한 열로 표시한다.
- 기본 화면에는 특성 수치, 장비 핵심 옵션 최대 2개, 보석 대상 스킬, 각인 레벨, 패시브·스킬 이름처럼 비교에 필요한 정보만 표시한다.
- 데스크톱의 주요 이름은 10~11px, 핵심 수치는 16px 이상을 유지하며 원문 설명은 카드 클릭 상세 창에서 확인한다.
- 장비의 별도 작은 버튼은 보이지 않게 하고 카드 전체를 클릭 가능한 상세 버튼으로 사용한다.
- `character.css` 변경 시 브라우저/CDN 캐시를 피하도록 `character.html`의 쿼리 버전도 함께 갱신한다.

### 주요 웹 파일

| 파일 | 역할 |
|---|---|
| `index.html` | 메인과 캐릭터 검색 진입 |
| `character.html` | 캐릭터 전용 화면과 정보 모듈 뼈대 |
| `character.js` | API 호출, 데이터 정규화, 모듈 렌더링, 탭과 상세 창 |
| `character.css` | 캐릭터 프로필·정보표·반응형 디자인 |
| `app.js` | 공통 메뉴, 정책 링크, 데스크톱 앱 배포 정보 |
| `styles.css`, `responsive.css` | 공통 디자인과 반응형 규칙 |

## 4. 백엔드 구현 현황

백엔드는 FastAPI 기반이며 서버 PC에서 `127.0.0.1:8600`으로 실행하고 Cloudflare Tunnel을 통해 `api.jloa.cloud`로 공개하는 구조다.

| 엔드포인트 | 동작 | 캐시 |
|---|---|---|
| `GET /api/health` | 서버 상태와 버전 | 없음 |
| `GET /api/roster/{name}` | 원정대 캐릭터 목록 | 3분 |
| `GET /api/roster/{name}/summary` | 같은 서버 원정대의 이름·직업·전투력 요약 | 원정대·캐릭터 각각 3분 |
| `GET /api/character/{name}` | 캐릭터 Armory 전체 정보 | 3분 |
| `GET /api/prices/refine` | 재련 재료 가격 | 15분 |

캐시 정책은 다음과 같다.

1. TTL 안이면 SQLite DB의 최근 값을 즉시 반환한다.
2. TTL이 지났으면 로스트아크 Open API를 조회하고 DB를 갱신한다.
3. Open API 조회가 실패하고 이전 DB 값이 있으면 `stale: true`로 이전 값을 반환한다.
4. 존재하지 않는 캐릭터도 3분간 음성 캐시한다.
5. 같은 캐릭터의 동시 갱신은 single-flight 잠금으로 한 번만 수행한다.
6. `?fresh=true`는 TTL을 무시하고 Open API를 즉시 조회한 뒤 DB에 저장한다.

원정대 기본 목록에는 전투력이 없으므로 요약 API는 같은 서버 캐릭터의 프로필을 최대 4개씩 병렬 조회한다. 기존 캐릭터 캐시를 재사용하며, 개별 프로필 조회가 실패하면 해당 행의 전투력만 `확인 불가`로 남기고 나머지 목록은 반환한다.

공통 응답 형태:

```json
{
  "data": {},
  "fetched_at": "ISO-8601 UTC",
  "cached": false,
  "stale": false,
  "fresh_requested": false
}
```

추가 보호 기능:

- IP당 30요청/분 슬라이딩 윈도 레이트리밋
- Cloudflare 사용 시 `CF-Connecting-IP` 기준
- 기본 CORS 허용 출처는 `jloa.cloud`, `www.jloa.cloud`, 로컬 개발 주소
- API 키는 서버의 `JLOA_API_KEY` 환경변수에서만 읽으며 프론트에 포함하지 않는다.

Windows에서 `setx` 또는 Machine 환경변수로 키를 등록한 뒤에는 기존 PowerShell과 실행 중인 서비스가 자동 갱신되지 않는다. 반드시 새 터미널 또는 새 서비스 프로세스에서 서버를 실행한다.

## 6. 아직 구현되지 않은 핵심 기능

메인에 표시된 다음 기능은 현재 안내 카드와 `준비 중` 토스트만 있고 실제 계산 기능은 없다.

- 스펙 시뮬레이터
- 스펙업 가이드
- 효율표

또한 현재 캐릭터 화면은 Open API 데이터를 자세히 정리해 보여주는 단계다. Lopec의 독자적인 스펙 점수나 효율 계산과 동등한 계산 엔진은 아직 구현되지 않았다.

추가로 확인할 운영 항목:

- `app.js`의 데스크톱 앱 버전과 파일명이 아직 `0.0.0` 기준이다.
- 실제 출시 전 설치 파일, SHA-256, VirusTotal 정보와 버전을 한 번에 갱신해야 한다.
- 웹 `README.md` 일부 안내가 현재 도메인·지원 링크 상태보다 오래되었으므로 추후 정리한다.
- `character.css`에는 디자인 변경 과정에서 누적된 중복 선택자와 후반 override가 많다. 현재 화면을 기준으로 시각 회귀 검증을 마련한 뒤 정리한다.

## 7. 다음 개발 권장 순서

1. 시뮬레이터에서 변경 가능한 항목과 계산 공식을 문서로 확정한다.
2. 백엔드 계산 입력·출력 JSON 스키마와 엔드포인트를 먼저 정의한다.
3. 실제 장비 데이터와 가상 변경값을 분리한 계산 엔진을 테스트 주도로 구현한다.
4. 스펙업 후보별 비용과 증가량을 계산해 가이드·효율표에서 공통 사용한다.
5. 캐릭터 페이지에는 새 기능을 별도 탭이나 모달로 연결해 현재 정보표의 밀도를 해치지 않게 한다.
6. API 호출량, 캐시 정책, 잘못된 툴팁 데이터와 누락 필드에 대한 회귀 테스트를 추가한다.

## 8. 작업 시 지켜야 할 사항

- 사용자가 요청하지 않은 `lostark_app` 데스크톱 앱 변경사항을 웹/백엔드 커밋에 섞지 않는다.
- 웹은 `jloa-web`, 백엔드는 `jloa-backend`에 각각 독립적으로 커밋하고 푸시한다.
- Open API 호출과 API 키 관리는 백엔드에서만 수행한다.
- 기존 3분 캐시 정책을 바꾸려면 호출량과 최신성 요구를 먼저 구분한다.
- 메인은 기능 개요와 검색 진입을 유지하고, 캐릭터 정보는 `character.html`에 유지한다.
- 모바일 가로 오버플로와 지나치게 작은 글씨가 다시 생기지 않도록 데스크톱·모바일을 모두 확인한다.
- 정적 CSS/JS를 변경하면 HTML의 캐시 버전 문자열도 함께 갱신한다.
- 외부 사이트는 레이아웃과 정보 분류의 참고 자료일 뿐이며 저작물이나 독점 로직을 그대로 가져오지 않는다.
